# Handoff — Fix CSV Import Dedup & Row Ordering

**Date**: 2026-07-09
**Goal**: Fix duplicate rows and incorrect ordering caused by flawed unique constraint on transactions table

## Context

User noticed duplicates in the DB (including a 2,059,995.98 withdrawal appearing twice) and that row ordering in the app doesn't match the bank's CSV order when multiple transactions share the same timestamp.

Investigation revealed:
- Kasikorn Bank CSV exports use TIS-620 encoding (app already handles this correctly via `parseBankCSV` in `src/lib/utils.js`)
- The unique constraint is `(tx_datetime, balance)` — defined in `supabase/migrations/20260522221703_remote_schema.sql` line 498
- The `import_transactions` RPC uses `ON CONFLICT (tx_datetime, balance) DO NOTHING`
- **Root cause**: The bank re-orders same-timestamp transactions between CSV downloads. This changes the running balance per row, so the same real-world transaction gets different balance values across exports → constraint doesn't catch it as duplicate
- Confirmed with IDs 6547/6587 (โอนเงินผ่าน SMART, 2,059,995.98) and IDs 6549/6586 (ฝากเช็คเรียกเก็บ, 44,479.76) — same transactions imported from two different CSV downloads (July 6 and July 8) with different balance columns

The app orders by `tx_datetime DESC, id DESC` (see `src/hooks/useTransactions.js` line 74-75). Since `id` is insertion order (not bank order), same-timestamp rows display in wrong sequence.

## Next Steps

1. **Write migration** adding `csv_row_order INTEGER` column to `transactions` table — stores row position from CSV (0-indexed within file, or within same-timestamp group)
2. **Change unique constraint** from `(tx_datetime, balance)` to `(tx_datetime, description, COALESCE(withdraw,0), COALESCE(deposit,0), channel)` — identifies transaction by what-it-is, not by running-balance which is ordering-dependent
3. **Update `import_transactions` RPC** to accept and store `csv_row_order`, use new conflict target, and add `DO UPDATE SET balance = EXCLUDED.balance, csv_row_order = EXCLUDED.csv_row_order` so re-imports correct stale balance values
4. **Update `parseBankCSV`** to include row index in output
5. **Update frontend ordering** — secondary sort by `csv_row_order DESC` instead of `id DESC`
6. **Data cleanup** — delete rows 6547 and 6549 (wrong balances from stale import), or let re-import with new ON CONFLICT DO UPDATE handle it automatically
7. **Test** with the sample CSV file (`Activity-26070823054418.csv`)

## Open Questions

- Should `ON CONFLICT` do `DO UPDATE` (overwrite balance/order from latest CSV) or `DO NOTHING`? User likely wants latest CSV to win since bank may correct data.
- Should we run a full dedup scan across all 4983 rows, or only fix the known 2 pairs?
- The proposed constraint `(tx_datetime, description, withdraw, deposit, channel)` — could there be legitimate distinct transactions with identical values in all these columns? (e.g., two separate "ฝากเช็คเรียกเก็บ" deposits of same amount at same second via same channel). If so, may need `cheque_number` in constraint too.
- Does user want balance column to always reflect latest CSV, or preserve original?
